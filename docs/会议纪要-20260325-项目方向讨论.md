# 会议纪要：鼎新项目方向讨论

> **日期**：2026年3月25日 16:11 - 约18:10（两段录音，合计约2小时）
> **参会人**：峻岭、何威、方拙、淼鑫、王瑾、家兵、龙飞、林超、文斌、自然（内部采购/运营）
> **核心议题**：鼎新项目是否从"跨平台比价"转向"基于天宫系统的药店AI经营助手"

## 一、会前各方调研汇报

### 方拙：药店场景调研
与外部药店从业者讨论后，按优先级排列药店痛点场景：
1. **比价**（最高频，每天都在做）
2. **断货补货提醒**
3. **返利管理**
4. **药品召回/效期管理**（到店后效期被罚、提前促销减损）
5. O2O市场情况（需美团等外部数据，短期难做）
6. 店员AI教练（非执业药师的营业员赋能）
7. 行业群甩货/采货信息聚合
8. 慢病管理

### 另一位同事：外部专家反馈
找外部人士做了批判性讨论，核心反馈：
- **身份问题**：药城本身是B2B交易平台，"运动员当裁判"，做入口的品牌定位需独立
- **数据挑战**：短期内推不动数据开放（数据分散在各B2B平台、ERP厂商、中康/米内网等手中）
- **网络效应门槛**：启动阶段难达到网络效应
- **建议定位**：不要一开始就做"操作系统"，先定位为**"药店AI买方经纪人"**——减少外部依赖，从采购切入，逐步扩展

---

## 二、核心讨论：三个切入方向PK

会议在白板上列出三个候选方向：

| 方向 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| **① 单品比价** | 当前在做的跨平台爬虫比价 | 用户每天都需要，刚需 | 技术难度大（爬虫反爬）、依赖外部平台 |
| **② AI采购（批量比价）** | 将原有AI采购改造为agent自然语言交互 | 已有后端能力，外部依赖最少 | 用户实际使用量不高，核心痛点不在比对而在数据收集 |
| **③ 智能补货/经营助手** | 基于天宫系统ERP数据，部署AI agent到药店 | **战略价值最大**——获取药店全量经营数据 | 需要药店同意接入ERP数据 |

### 关键洞察

**自然（内部采购）分享的真实采购流程**：
- 采购员从蜂鸟BI看库存报表，每个采购的补货规则都不同（有的看7天日均销量，有的看28天）
- 蜂鸟只提供原始数据宽表，没有智能推荐，采购经验没有统一沉淀
- 补货清单有了，但采购资源不一定到位（找不到货、价格不合适）

**关于AI采购现状的诊断**（方拙）：
- 当前用户主要上传的是线上平台报价，很少带线下供应商
- 用户核心痛点不是"汇总后选谁便宜"，而是**"从各平台拿数据这个过程本身就很长"**
- 内部集采团队也不常用，说明交互改造不能解决根本问题
- 连锁采购实际行为：大供应商（5-6家）会批量比价，但剩下40-50家窜货商业公司**只和历史采购价对比**，不做跨供应商比较

## 三、核心决策：战略转向

### 核心判断

> "我们要的就是个入口。OpenClaw真正牛的地方不是它能干什么，是它把入口抓到手里。"

> "没有什么战略价值比得上——我能读他所有的数据。如果我有10万家药店的库存数据，每家药企都需要这个东西。"

### 达成共识

**方向③（基于天宫的AI经营助手）为最高优先级**，理由：
1. **天宫系统已有基础**：已接入数千家药店ERP，存量数据30-40TB，具备库存+销售+采购数据
2. **战略价值最大**：掌握药店全量经营数据 → 对上游药企议价能力 → 网络效应
3. **先占先赢**：部署后替换成本极高（用户习惯+数据沉淀）
4. **启动速度快**：不依赖爬虫等外部能力，利用现有数据通路

## 四、决策事项

| # | 决策 | 结论 |
|---|------|------|
| 1 | **OpenClaw部署方式** | 云端部署（不在天宫本地扩展），最快验证 |
| 2 | **入口定位** | 确认做"入口"，大共识不变 |
| 3 | **项目优先级调整** | 天宫+OpenClaw经营助手为**最高优先级**；比价/爬虫降级，不停但不再压时间 |
| 4 | **计价体系** | 需建立token/credit换算的计价模型（成本加价，毛利目标待定，峻岭倾向高毛利） |
| 5 | **目标** | **两周内首批用户上线**（不少于5家外部 + 内部药店） |

## 六、技术路径初步共识

```
药店ERP（已有天宫插件接入）
    │
    │ 同步库存+销售+采购数据
    ▼
数据层（待定：原始数据 or 大数据团队清洗后的数据）
    │
    │ API接口
    ▼
OpenClaw（云端部署）
    │
    ├── Skill 1：补货提醒（优先）
    ├── Skill 2：效期预警/临期促销退货
    ├── Skill 3：经营数据分析（口喷式）
    ├── Skill 4：MOR/WBR 经营周报月报
    └── ... 更多Skill持续扩展
    │
    ▼
药店老板/采购员（对话交互）
```

**关键技术问题待明日确认**：
- 天宫同步的数据是实时的还是有延迟？单体 vs 连锁的数据完整度差异？
- 数据传给OpenClaw的方式：直接传原始数据（token消耗大）vs 封装API接口调用（推荐）
- Skill设计思路：不要过度定义业务规则，先提供数据访问能力 + 几个打样Skill，让用户自己探索

---

## 七、其他要点

- **Token成本控制**：大量数据分析不应直接用大模型处理，应将重复逻辑沉淀为Skill/脚本（规则处理不消耗token），大模型只处理理解和推理
- **比价项目不停**：林超爬虫已基本完成，产品在调试阶段，作为一个Skill保留，但时间紧迫度降低
- **竞争威胁**：头部ERP公司（有client base）可能做类似的事，需要抢先占位
- **商业模式雏形**：token reseller模式——免费给药店用（每日限额），培养依赖后收费
