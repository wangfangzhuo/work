# 药城版 OpenClaw 落地方案

## 一、我们有什么

先说清楚我们手里已经有的东西，这决定了我们能做什么：

```
┌─────────────────────────────────────────────────┐
│  天宫系统（已有）                                  │
│                                                   │
│  直连药店 ERP → 云服务器存储以下数据：              │
│                                                   │
│  · 采购数据：从哪买、买了什么、多少钱、什么时候买     │
│  · 库存数据：有什么货、多少量、效期、批号             │
│  · 销售数据：卖了什么、卖多少、什么价格、卖给谁       │
│  · 可能还有：供应商信息、客户信息、财务数据等         │
└─────────────────────────────────────────────────┘
```

**这意味着什么？**

之前我们讨论的方案，最大的难点是"怎么拿到药店的数据"。现在这个问题不存在了——天宫系统已经帮我们拿到了。我们要做的只是：**在这些数据之上，套一层自然语言交互的 AI 能力。**

---

## 二、我们要做什么

一句话：**让药店老板用说人话的方式，从自己的经营数据里获得决策建议。**

不需要他会看报表，不需要他会用 Excel，他只需要说：

> "帮我看看哪些药快过期了"
> "这个月采购花了多少钱？比上个月多还是少？"
> "帮我生成下周的采购清单建议"
> "我的库存结构健不健康？"
> "哪些药卖得好但库存不够了？"

Agent 直接从天宫的数据里查出来，分析完，给他结论和建议。

## 四、核心需要做的事（按优先级排）

### 第一件事：建数据中间层（后端，约1-2周）

**为什么需要这一层？**

天宫系统里存的是原始数据——几百张表、几千个字段，AI 没法直接读。数据中间层的作用是：**把"数据库里的原始数据"变成"AI 能理解和使用的结构化接口"。**

**类比**：天宫系统像一个巨大的图书馆，但没有目录。数据中间层就是帮图书馆编一个目录，让 AI 分析师能快速找到他需要的书。

**具体做什么**：

```
举例：效期查询接口

输入：药店ID + 时间范围（比如3个月内过期）
输出：
{
  "store_id": "S001",
  "expiring_items": [
    {
      "drug_name": "阿莫西林胶囊",
      "manufacturer": "联邦制药",
      "spec": "0.5g×24粒",
      "batch": "20240315",
      "expiry_date": "2026-06-15",
      "remaining_days": 82,
      "quantity": 35,
      "unit_cost": 12.5,
      "total_value": 437.5
    },
    ...
  ],
  "summary": {
    "total_items": 47,
    "total_value": 23000,
    "urgent_count": 8,       // 30天内过期
    "warning_count": 15,     // 30-60天内过期
    "attention_count": 24    // 60-90天内过期
  }
}
```

**需要建的核心接口**（V1先做这几个，够用就行）：

| 接口 | 作用 | 对应用户问题 |
|------|------|-------------|
| 效期查询 | 按紧急程度列出即将过期的药品 | "哪些药快过期了" |
| 库存概览 | 库存结构、周转率、积压品种 | "我的库存健不健康" |
| 采购记录 | 按时间/品种/供应商查采购明细 | "这个月采购花了多少" |
| 销售排行 | 热销/滞销品种、销售趋势 | "哪些药卖得好" |
| 补货建议 | 基于销售速度和库存量计算补货清单 | "帮我生成采购清单" |
| 经营摘要 | 关键经营指标一览 | "我这个月经营情况怎么样" |

### 第二件事：开发 Skill（AI 层，约1-2周，可与第一件事并行）

**Skill 是什么？**

Skill 是 OpenClaw 里的一个个"技能模块"。每个 Skill 负责处理一类问题。AI 大脑（Claude）负责理解用户说了什么，然后调用对应的 Skill 去执行。

**类比**：AI 大脑是一个全科医生，Skill 是各个科室的专科医生。全科医生听完你的症状，判断该转给哪个科室，然后专科医生出具体方案。

**V1 先做这几个 Skill**：

#### Skill 1：效期预警（优先级最高）

```
用户说："帮我看看哪些药快过期了"

Skill 执行流程：
1. 从用户身份获取药店ID
2. 调用数据中间层的效期查询接口
3. 按紧急程度分三档（30天/60天/90天）
4. 计算总损耗金额
5. 给出处理建议（可退货的退货、可促销的促销、要报损的报损）

输出给用户（对话卡片形式）：
┌─────────────────────────────────────┐
│  ⚠️ 效期预警报告                      │
│                                     │
│  3个月内过期：47个品种，货值 ¥23,000   │
│                                     │
│  🔴 紧急（30天内）：8个品种 ¥6,200    │
│  🟡 预警（30-60天）：15个品种 ¥9,800  │
│  🟢 关注（60-90天）：24个品种 ¥7,000  │
│                                     │
│  建议：                              │
│  · 3个品种可联系供应商退货（¥2,100）   │
│  · 5个品种建议促销清货                │
│  · 查看详细清单 →                     │
└─────────────────────────────────────┘
```

#### Skill 2：采购清单生成

```
用户说："帮我生成下周的采购清单"

Skill 执行流程：
1. 获取近30天销售数据，算出各品种日均销量
2. 获取当前库存量
3. 计算：预计几天卖完 = 当前库存 ÷ 日均销量
4. 筛出7天内会缺货的品种
5. 计算建议采购量 = 日均销量 × 备货天数 - 当前库存
6.（如果有比价能力）推荐采购渠道和价格

输出：一份可下载的采购建议清单
```

#### Skill 3：库存健康诊断

```
用户说："我的库存结构健不健康？"

Skill 执行流程：
1. 获取全量库存数据
2. 交叉销售数据，计算每个品种的周转天数
3. 分类：
   - 高周转（<30天）= 健康
   - 中周转（30-90天）= 正常
   - 低周转（>90天）= 积压
   - 零销售 = 死库存
4. 计算积压资金占比
5. 给出优化建议

输出：库存健康度评分 + 积压品种清单 + 优化建议
```

#### Skill 4：经营日报/周报

```
用户说："帮我看看这周经营情况"

Skill 执行流程：
1. 获取本周销售、采购、库存数据
2. 计算关键指标：销售额、毛利率、采购支出、库存周转
3. 与上周/上月同期对比
4. 标出异常项（比如某品种销量突然下降）

输出：结构化的经营摘要卡片
```

## 五、数据流转（一个完整的用户请求是怎么跑的）

以"帮我看看哪些药快过期了"为例，完整流程：

```
第1步：用户在聊天框输入 "帮我看看哪些药快过期了"
  │
第2步：OpenClaw 收到消息，调用 AI 大脑（Claude）
  │    AI 理解意图 → 这是一个效期查询请求
  │    AI 决定 → 调用"效期预警 Skill"
  │
第3步：效期预警 Skill 开始执行
  │    ├─ 从用户登录信息获取药店ID
  │    └─ 调用数据中间层 /api/expiry?store_id=S001&days=90
  │
第4步：数据中间层收到请求
  │    ├─ 查询天宫系统数据库
  │    ├─ SQL: SELECT * FROM inventory WHERE expiry_date < now()+90days
  │    └─ 返回结构化数据（47个品种，总值23000）
  │
第5步：Skill 拿到数据，进行业务处理
  │    ├─ 按紧急程度分三档
  │    ├─ 计算可退货/可促销的品种
  │    └─ 生成处理建议
  │
第6步：AI 大脑组织语言 + 生成卡片
  │    ├─ 文字："您的店里有47个品种将在3个月内过期..."
  │    ├─ 卡片：效期预警报告（结构化数据）
  │    └─ 追问："要不要看看补货建议？"
  │
第7步：前端展示给用户（对话气泡 + 卡片）
```

## 六、技术选型建议

| 模块 | 建议方案 | 理由 |
|------|---------|------|
| AI 大脑 | Claude（通过 OpenClaw） | 已有 OpenClaw 基座，中文理解能力强 |
| Skill 开发 | OpenClaw Skill 框架 | 团队已在学习，与现有架构一致 |
| 数据中间层 | RESTful API（Node.js 或 Python） | 轻量，开发快，团队熟悉 |
| 数据库查询 | 直接查天宫的数据库 / 建轻量数据视图 | 不搬数据，减少同步问题 |
| 前端 | 现有 Next.js + 对话组件 | 复用 pharma-compare 前端 |
| 卡片渲染 | 自定义组件（React） | 效期卡片、采购清单等需定制 |
| 用户认证 | 对接天宫/111用户体系 | 药店ID是所有查询的起点 |

## 七、分阶段落地计划

### Phase 1：最小可用版（3周）

**目标**：一个能跑的 demo，覆盖效期预警 + 采购建议两个场景。

| 周次 | 事项 | 负责 |
|------|------|------|
| 第1周 | 数据中间层：效期查询 + 库存查询 + 销售查询三个接口 | 后端 |
| 第1周 | Skill 开发：效期预警 Skill + 采购清单 Skill | 产品+后端 |
| 第1周 | 前端：对话界面 + 效期卡片 + 采购清单卡片 | 前端 |
| 第2周 | 联调：OpenClaw → Skill → 数据接口 全链路跑通 | 全员 |
| 第2周 | 接入真实药店数据测试 | 产品+后端 |
| 第3周 | 修 bug + 优化体验 + 内部试用 | 全员 |

**Phase 1 交付物**：
- 药店老板打开 H5，登录后说"帮我看看哪些药快过期了"，立刻看到自己店里的效期预警报告
- 说"帮我生成采购清单"，得到基于真实销售数据的补货建议

### Phase 2：场景扩展（第4-6周）

- 库存健康诊断 Skill
- 经营日报/周报 Skill
- 销售趋势分析 Skill
- 接入跨平台比价能力（采购清单 → 哪家更便宜）

### Phase 3：决策闭环（第7-9周）

- 从"给建议"到"帮执行"：采购清单 → 一键下单
- 主动推送：不等用户问，Agent 主动发效期预警、缺货提醒
- 多药店支持：连锁药店总部看所有门店汇总

## 八、这个方案对之前讨论的影响

有了天宫系统的数据，之前讨论的策略可以直接落地：

| 之前的设想 | 现在怎么做 |
|-----------|-----------|
| "让他看见自己在亏钱" | 用户登录就能看到效期损耗金额——零门槛 |
| "信任三步走" | 第一步直接跳过数据获取，进入价值展示 |
| "第一刀用效期做钩子" | 天宫有库存和效期数据，可以直接做 |
| "比价做留存" | 效期预警 → 补货 → 自然引出比价 |
| "经营决策代理" | 天宫的数据覆盖了采购/库存/销售，全链路数据都有 |

**天宫系统让我们跳过了最难的冷启动阶段**——我们不需要说服药店老板"先把数据给我"，因为数据已经在了。我们只需要证明"你的数据在我手里能帮你赚钱/省钱"。

## 九、工作量分析

**工作量最大的是数据中间层，不是 AI 层。**

| 模块 | 表面工作量 | 实际工作量 | 原因 |
|------|-----------|-----------|------|
| 数据中间层 | 看起来就是写几个 API | ★★★★★ | 天宫数据库的复杂度在这里爆发 |
| Skill 开发 | 看起来要写很多业务逻辑 | ★★★ | 逻辑本身不复杂，难在依赖数据层的质量 |
| OpenClaw 对接 | 看起来简单 | ★★★ | 意图识别的准确率需要反复调 |
| 前端 | 看起来就是个聊天框 | ★★ | 有现成组件，卡片定制量有限 |

数据中间层为什么最重：

1. **天宫的数据库不是为 AI 设计的**——ERP 系统的表结构是为业务流程设计的，可能几十上百张表、字段命名不规范、关联关系复杂。光搞清楚"效期"这个信息需要关联哪几张表、取哪些字段，可能就要跟天宫的开发花好几天对齐
2. **不同药店的数据格式可能不统一**——有的药店录入规范，有的乱写；有的有批号效期，有的没录；药品名称可能不标准
3. **查询性能**——AI 需要实时返回结果，但天宫数据库可能没有为这种查询场景做索引优化

## 十、风险矩阵

### 🔴 高风险（大概率发生 + 影响大）

**风险1：数据质量问题**

- **表现**：药店 ERP 数据缺失、错误、不一致。比如效期没录、库存数量和实际不符、采购价格是0
- **概率**：高。ERP 数据质量差是行业通病，药店老板或店员录入不规范是常态
- **影响**：Agent 说"47个品种快过期"但实际只有20个——老板验证一次就不信了。**数据不准，信任链直接断裂**
- **应对**：
  - V1 先选数据质量最好的几家药店做试点
  - 输出标注数据置信度——"本报告基于 ERP 库存数据，如有出入请以实际盘点为准"
  - 做数据质量评分，数据太差的药店先不开放

**风险2：天宫系统的配合度和接口稳定性**

- **表现**：天宫团队排期紧配合不了、接口文档不全反复沟通、数据库结构变更导致接口挂掉
- **概率**：高。天宫有自己的迭代节奏，跨团队协作永远比预期慢
- **影响**：数据中间层完全依赖天宫的数据，天宫不配合就什么都做不了。**项目最大的外部依赖**
- **应对**：
  - 第一周就派人跟天宫团队坐在一起，搞清楚数据库结构和接口能力
  - 争取天宫数据库的只读权限，我们直接查，不等他们建新接口
  - 准备 mock 数据，让 Skill 开发不被数据层阻塞

### 🟡 中风险（可能发生 + 影响中等）

**风险3：AI 意图理解不准**

- **表现**：用户说"帮我看看这个月花了多少钱"，AI 分不清是采购支出还是全部经营支出
- **应对**：拿不准时让 Agent 反问确认；V1 先限定支持的问题类型，做好做透不贪多

**风险4：OpenClaw Skill 约束能力不足**

- **表现**：团队日报已反馈 Skill 约束能力弱，无法作为严谨工作流
- **应对**：数据查询核心逻辑用代码编排，Skill 只负责理解意图和组织输出。**让 AI 做理解人话，让代码做查数据算数字**

**风险5：响应速度慢**

- **表现**：链路太长（用户→OpenClaw→AI→Skill→数据中间层→天宫数据库），每个环节都有延迟
- **应对**：高频查询做缓存（效期数据每天更新一次即可）；先返回"正在分析"的状态提示，流式输出结果

### 🟢 低风险但需注意

- **数据安全合规**：每个查询严格校验药店ID权限，A 药店不能看到 B 药店数据
- **用户预期管理**：明确能力边界，超范围问题优雅拒绝

---

## 十一、最该优先验证的两件事

| 排序 | 风险 | 一句话 | 建议行动 |
|------|------|--------|---------|
| **1** | 数据质量 | 数据不准，信任链直接断 | 第一周拿一家真实药店的天宫数据跑一遍效期查询，看数据能不能用 |
| **2** | 天宫配合度 | 最大外部依赖，卡住就全停 | 第一周跟天宫团队确认数据库访问方式和权限 |

**这两个结论拿到了，项目才真正能动。否则后面所有的 Skill 开发、前端开发都是空中楼阁。**
