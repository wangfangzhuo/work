## 一、Skill 调用链路

### 完整链路

```
用户第一次打开对话
        ↓
Skill: check_auth
检查用户授权了哪些平台
  → 若有平台未授权，Agent 主动提醒：
    "您还没有授权小药药，授权后可以多一个平台的比价数据"

用户说"帮我查阿莫西林的价格"
        ↓
Skill: get_user_history
查用户在我们系统内的历史查询和采购记录
（用于对搜索结果排序）
        ↓
Skill: search_drug
在所有已授权平台搜索药品，返回候选列表
  → 结合历史数据 + 平台热度排序，最可能的选项排在前面
  → Agent 展示给用户："请问您要查哪个？"
        ↓
用户说"第一个"
        ↓
Skill: match_drug
将各平台返回的商品映射到内部主数据，判定是否为同一个品
        ↓
Skill: get_price（并行）
同时向所有已授权平台查这个药的价格，谁先回来先展示谁
        ↓
Agent 把各平台价格整理成比价卡片展示给用户
```

### Skill 列表

| Skill | 干什么 | 谁来做 |
|-------|-------|-------|
| check_auth | 检查并管理平台授权状态，首次进入时触发 | 家兵 |
| get_user_history | 查用户历史查询/采购记录，用于搜索结果排序 | 家兵 |
| search_drug | 在各已授权平台搜索药品候选列表 | 林超 |
| match_drug | 外部商品映射到内部主数据码，判定是否同一个品 | 家兵（对接内部对码系统） |
| get_price | 并行查各平台价格，先返回先展示 | 林超 |

## 二、药品匹配策略

各平台药品资料写法不统一（如"10g\*9袋"vs"10克\*9袋"），解决方案：

**匹配优先级：**

```
① 内部对码系统（主路径，覆盖率95%+）
  各平台商品 → 对码 → 映射到同一个内部主数据码 → 同码即同品
        ↓ 对不上内部码时
② 字段完全一致匹配
  名称 + 规格 + 厂家 三个字段完全相同 → 认定为同品
        ↓ 以上都对不上时
③ 展示给用户手动选择，不强行合并
```

**竞争壁垒**：内部对码系统覆盖率95%+，是外部竞品短期内难以复制的能力，是鼎新相对竞品的隐性壁垒，可在内部汇报中单独提出。

**3/20 技术对齐需确认**：内部对码系统接口已有，家兵直接对接封装成 match_drug Skill 即可。

## 三、搜索入口与导流设计

### 入口形态

**触发时机**：用户在1药城搜索框发起搜索时触发

**PC端**：搜索结果页右侧弹出对话框

| 用户状态 | 弹窗内容 | 操作按钮 |
|---------|---------|---------|
| 未授权 | "「鼎新」支持跨平台比价，授权您的采购平台账号即可使用" | [去授权] [关闭] |
| 已授权 | "检测到您搜索了「XX」，要帮您跨平台比价吗？" | [开始比价] [不用] |

**移动端**：底部抽屉（侧边空间不足，改为从底部弹出）

**手动触发**：用户关闭弹窗后，页面右下角保留**悬浮球**，可随时手动唤起对话

### 弹窗出现规则

| 用户状态 | 关闭后行为 |
|---------|-----------|
| 未授权 | 关闭后30天内不再主动弹出，30天后恢复提醒 |
| 已授权 | 点"不用"后本次会话不再弹，悬浮球保留 |

### 平台优先级

V1 优先 PC Web，同步兼容移动端
