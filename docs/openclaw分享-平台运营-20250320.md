# OpenClaw 分享会 — 平台运营场景与思考

---

## 一、理解 OpenClaw：它到底是什么

**OpenClaw 不是一个 AI 聊天机器人，而是一个"AI 能力装配平台"**

核心概念：

```
┌──────────────────────────────────────────────────┐
│                   OpenClaw 平台                    │
│                                                    │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐          │
│   │ Skill A │  │ Skill B │  │ Skill C │  ← 可插拔的能力单元  │
│   └────┬────┘  └────┬────┘  └────┬────┘          │
│        │            │            │                │
│   ┌────┴────────────┴────────────┴────┐          │
│   │        AI Agent（智能体）          │          │
│   │  理解意图 → 选择 Skill → 执行 → 反馈 │          │
│   └───────────────────────────────────┘          │
│        │                                          │
│   ┌────┴──────────────────────────────┐          │
│   │     企业内部系统 API / 数据         │          │
│   └───────────────────────────────────┘          │
└──────────────────────────────────────────────────┘
```

- **Skill**：一个封装好的能力单元（比如"活动商品自动审批"、"资源位排期推荐"）
- **Agent**：理解用户意图，调用合适的 Skill 来完成任务
- **平台**：提供 Skill 的开发、部署、权限、监控、灰度等基础设施

**为什么 OpenClaw 适合解决运营的问题：**

| 能力 | 说明 | 对运营的意义 |
|------|------|------------|
| **直连内部系统** | Skill 可以直接调用后台 API（活动配置、资源位管理、商品系统） | 不只是问答，能真正帮运营操作系统 |
| **权限管控** | 与内部权限体系打通，Skill 级别可控 | 谁能审批、谁能配置，和现有权限一致 |
| **业务规则封装** | 可以把活动规则、审批标准、SOP 深度封装进 Skill | 运营经验变成可执行的规则，不再依赖个人 |
| **可沉淀可迭代** | Skill 是公司级资产，可复用、可传承 | 团队能力不因人员流动而流失 |
| **自动执行** | Skill 可以按规则自主执行，不需要人工逐步操作 | 从"人操作系统"变成"人定规则，Skill 执行" |

## 二、平台运营：三大职责与 Skill 化全景

平台运营三大核心职责：

1. **活动发布与配置**：发布平台活动，配合 JBP/MP/采购等团队进行商品提报和配置维护
2. **资源位管理**：规划资源位排期，跨团队协商需求，完成配置（类型多、需求大、协作复杂）
3. **数据追踪与行动建议**：追踪日常数据和活动数据，评估各项目效果，给出后续行动项

### Skill 化覆盖全景

| 职责 | 可 Skill 化的场景 | 当前痛点 |
|------|-----------------|---------|
| **活动发布与配置** | 商品提报自动筛选、跨团队提报汇总、提报自动审批、满减/券批量配置、上线自动校验 | 手动提报+逐条审批+逐个配置，每次大促重复一遍 |
| **资源位管理** | 需求自动汇总、冲突识别、排期推荐、资源位内容配置、效果追踪 | Excel 排期+群里协商，类型多需求方多冲突频发 |
| **数据追踪** | 日常数据自动播报、活动效果自动复盘、异常指标预警、行动建议生成与推送 | 手动拉数据做表格写分析，耗时长且依赖个人能力 |

**核心矛盾：** 运营 70%+ 时间花在配置、协调、搬数据上，该做的策略思考反而没时间做。

## 三、重点场景：活动配置自动化

### 当前的活动配置流程

```
运营策划活动
    ↓
群里通知 JBP/MP/采购各团队 → 等各团队提报商品 → 反复沟通确认
    ↓
运营逐条审批提报（不合规的手写理由驳回）
    ↓
审批通过后，运营在后台逐个商品配置满减/券/打标
    ↓
上线前人工逐项检查配置是否正确
    ↓
上线后人工盯数据
```

**每次大促都重复这套流程，每次都靠人力堆。**

### OpenClaw 之后的流程

```
运营定义活动规则（门槛、满减机制、目标 GMV）
    ↓
[Skill] 按规则自动筛选商品 → 推送给各团队确认/补充提报
    ↓
[Skill] 各团队提报自动汇总 → 合规的秒批 → 驳回的自动附整改建议
    ↓
[Skill] 审批通过的商品 → 调用后台 API 批量完成满减配置与打标
    ↓
[Skill] 上线前自动校验配置一致性 → 异常项推送运营确认
    ↓
[Skill] 上线后自动巡检 → 异常主动预警
```

### 对比

| 环节 | 当前（人工） | OpenClaw 之后 |
|------|------------|--------------|
| 商品筛选 | 导数据、Excel 筛 | Skill 按规则自动筛选 |
| 跨团队沟通 | 群里反复@确认 | Skill 自动汇总，冲突自动标注 |
| 提报审批 | 逐条人工审核 | 合规秒批，驳回自动附理由 |
| 满减配置 | 后台逐个商品配置 | Skill 调 API 批量完成 |
| 上线检查 | 人工逐项核查 | Skill 自动校验，异常标红 |

## 四、产品支持模式的根本转变

**这是我认为最重要的一页——OpenClaw 不只是提效工具，而是改变了产品支持业务的方式**

### 传统模式：每次迭代都依赖产品和开发

```
业务有新需求（如：新增一种活动类型的审批规则）
    ↓
业务提需求给产品 → 产品写 PRD → 开发排期 → 开发 → 测试 → 上线
    ↓
下次规则变了 → 又提需求 → 又排期 → 又开发……
    ↓
周期长、响应慢、产品和开发疲于奔命，业务也等不及
```

### OpenClaw 模式：产品搭底座，业务自己迭代

```
产品 + 技术：搭建 1-2 个基础 Skill（如"活动商品审批 Skill"）
    ↓
    ├── 业务可以自己做的：修改审批规则、调整筛选条件、更新话术模板
    │   → 运营直接在 Skill 配置里改，不需要产品和开发介入
    │
    └── 业务做不了的：需要新的系统接口（如对接新的活动类型 API）
        → 产品提给技术，技术封装成新的底座能力
        → 封装完成后，运营又可以基于新能力自己组装 Skill
```

### 三个角色的分工

| 角色 | 职责 | 频率 |
|------|------|------|
| **产品团队** | 搭建 1-2 个标杆 Skill，跑通完整流程，沉淀模板和规范 | 一次性（+ 偶尔新场景） |
| **技术团队** | 封装底座能力——把内部系统接口包装成 Skill 可调用的标准 API | 按需（产品提出时） |
| **运营团队** | 基于已有 Skill 和底座能力，自己定义规则、调整参数、组装新的 Skill | 高频、自主 |

### 为什么这样更好

| 对比 | 传统模式 | OpenClaw 模式 |
|------|---------|-------------|
| 新活动规则上线 | 提需求→排期→开发→2-4 周 | 运营改 Skill 配置→当天生效 |
| 产品团队精力 | 疲于实现碎片化需求 | 聚焦底座能力和标杆场景 |
| 开发团队精力 | 反复开发相似功能 | 封装通用 API，一次开发多次复用 |
| 业务响应速度 | 排期等 2 周起步 | 简单调整即时生效 |
| 能力传承 | 在个人经验和代码里 | 在 Skill 规则里，可复用可传承 |

**一句话：产品从"帮业务实现每一个需求"变成"帮业务搭建自己解决问题的能力"。**
