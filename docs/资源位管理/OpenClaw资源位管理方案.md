# OpenClaw 落地场景：资源位全流程智能化

## 一、背景与痛点

### 平台运营的核心工作

平台运营日常需要对接 MP 商家运营、JBP 商家运营、厂牌精选运营、销售运营等多个团队，负责平台资源位的排期规划、配置上线和数据监控。

### 一次完整的资源位配置 = 15 个环节

```
① 业务方提需求（企微沟通）
    ↓
② 查排期、推荐资源位
    ↓
③ 确认排期，锁定资源位
    ↓
④ 沟通活动细节（主题、优惠券、商品、客户范围）
    ↓
⑤ 创建优惠券
    ↓
⑥ 提设计需求（给设计师）
    ↓
⑦ 设计师出图
    ↓
⑧ 收集活动商品
    ↓
⑨ 配置落地页（maps：头图 + 优惠券 + 导入商品）
    ↓
⑩ 配置入口（标题、封面图、链接、时段、客户范围）
    ↓
⑪ 上线前检查
    ↓
⑫ 提交审批（内部系统审批通过后才可上线）
    ↓
⑬ 上线
    ↓
⑭ 数据监控
    ↓
⑮ 效果反馈与优化建议
```

### 核心矛盾

- 单个环节都不复杂，但 **15 个环节串起来，涉及 4 个角色（业务方、平台运营、设计师、采购）、4 个系统（腾讯文档、后台配置、maps、神策）**
- 平台运营 70%+ 时间花在执行操作上，没有精力做策略优化
- 每周四沟通下周排期，涉及多个团队的需求汇总与冲突协调
- 数据效果监控靠手动拉取，基本没有后续优化动作

## 二、OpenClaw 改造方案

### 改造前 vs 改造后

| 维度 | 当前 | OpenClaw 之后 |
|------|------|--------------|
| 一次配置耗时 | 2-3 小时（分散在多天） | 3 次确认，执行全自动 |
| 运营角色 | 执行者（逐个操作 15 个环节） | 决策者（确认排期、确认商品、确认上线） |
| 数据监控 | 手动拉、偶尔做 | 每日自动播报 + 异常预警 + 优化建议 |
| 效果反馈 | 基本没做 | 自动生成效果报告推送给业务方 |

### 理想流程演示

```
平台运营截图业务方的企微消息丢给 AI
  ↓
[Skill] "我识别到以下需求：
  品牌：九芝堂
  期望时间：下周二至周四
  偏好：首页、流量大

  查询排期后推荐：
  1. 首页中通2（周二至周四全空，日均点击 XXX）✅ 推荐
  2. 首页弹窗（周三周四空闲，日均点击 XXX）
  3. 首页中通1（已被本地省心购占用）❌

  是否锁定首页中通2？"
  ↓
运营："锁定，活动是满199减30，商品用九芝堂的感冒灵系列"
  ↓
[Skill] 自动执行：
  ✅ 更新腾讯文档排期表 → 首页中通2 周二至周四 = 九芝堂
  ✅ 创建优惠券 → 满199减30
  ✅ 生成设计需求单 → 推送给设计师
     "九芝堂品牌推广，首页中通2入口图（750x360px）
      + 落地页头图（750x420px），周一下班前交付"
  ✅ 推荐商品列表 → 九芝堂感冒灵相关 12 个 SKU
  ↓
运营确认商品列表 ✅，设计师交付图片 ✅
  ↓
运营："图片好了，商品确认，上线吧"
  ↓
[Skill] 自动执行：
  ✅ 创建落地页（maps）→ 头图 + 优惠券 + 导入商品
  ✅ 配置入口 → 标题、封面图、链接、时段、客户范围
  ✅ 上线前校验 → 图片、链接、时段、商品库存全部通过
  ✅ 提交审批 → 推送到内部审批系统，附配置摘要和预览
  ↓
  ⏸ 等待审批（人工审批）
  ↓
  审批通过 → [Skill] 自动发布上线
  审批驳回 → [Skill] 通知运营驳回原因，提示修改项
  ↓
[Skill] 上线后每日自动：
  ✅ 拉取神策数据 → 生成日报
  ✅ 异常预警 → 点击率低于均值自动提醒
  ✅ 活动结束 → 自动生成效果总结推送给业务方
```

**平台运营从"执行 15 个环节"变成"确认 3 次 + 1 次审批"。**

## 三、15 个环节逐一拆解

| # | 环节 | 当前做法 | OpenClaw 之后 | 自动化程度 |
|---|------|--------|-------------|:---------:|
| ① | 业务方提需求 | 企微群聊沟通 | 截图识别 → 自动提取品牌、时间、偏好等结构化信息 | 全自动 |
| ② | 查排期+推荐资源位 | 看腾讯文档排期表，凭经验推荐 | 自动读取排期表 + 按历史流量排序推荐 + 生成回复话术 | 全自动 |
| ③ | 锁定排期 | 手动更新腾讯文档 | 运营确认后自动更新排期表 | 全自动 |
| ④ | 沟通活动细节 | 企微反复沟通 | AI 生成确认清单（主题？券面额？商品范围？），运营一次性发给业务方确认 | AI 辅助 |
| ⑤ | 创建优惠券 | 后台手动创建 | 运营告诉 AI 券的规则，自动调 API 创建 | 全自动 |
| ⑥ | 提设计需求 | 在线表格提需求给设计师 | 基于需求信息自动生成设计需求单，推送给设计师 | 全自动 |
| ⑦ | 设计师出图 | 设计师排期制作 | AI 不介入，等待设计师交付 | 人工 |
| ⑧ | 收集活动商品 | 业务方提供 Excel 或运营整理 | AI 按品牌+活动类型自动推荐商品列表，确认后导出 | AI 辅助 |
| ⑨ | 配置落地页 | maps 手动搭建（头图+券+商品） | 自动调 maps API：上传头图、挂载优惠券、导入商品 | 全自动 |
| ⑩ | 配置入口 | 后台逐个填字段 | 自动调后台 API 填充所有字段 | 全自动 |
| ⑪ | 上线前检查 | 人工逐项核查 | 自动校验：图片是否上传、链接是否可访问、时段是否正确、商品是否有库存 | 全自动 |
| ⑫ | 提交审批 | 人工提交 | 自动提交审批单到内部系统，附配置摘要+预览链接 | 全自动 |
| ⑬ | 上线 | 审批通过后手动发布 | 审批通过自动发布；驳回自动通知运营 | 全自动 |
| ⑭ | 数据监控 | 手动从神策拉数据 | 每日自动拉取，生成日报+异常预警 | 全自动 |
| ⑮ | 效果反馈 | 基本没做 | 自动生成效果报告+优化建议，推送给业务方 | 全自动 |

**15 个环节中：11 个全自动，2 个 AI 辅助，2 个人工（设计出图 + 审批）。**

## 四、Skill 设计

### 一个 Skill，内部 5 个模块

```
资源位全流程管理 Skill
├── 模块1：需求识别（截图识别 → 结构化提取）
├── 模块2：排期推荐（查排期 → 流量排序 → 推荐+话术）
├── 模块3：配置执行（优惠券+落地页+入口 → 一键配置）
├── 模块4：审批上线（自动提交审批 → 监听结果 → 发布/通知）
└── 模块5：数据监控（日报播报 → 异常预警 → 效果总结）
```

## 五、需要技术封装的底座 API

| # | API | 说明 | 优先级 |
|---|-----|------|:-----:|
| 1 | 腾讯文档读写 API | 读取/更新排期表（通过腾讯文档开放 API，OAuth 授权） | P0 |
| 2 | 神策数据查询 API | 按资源位+时间维度查询点击、访问、转化数据 | P0 |
| 3 | 优惠券创建 API | 创建满减/折扣券（面额、门槛、有效期、使用范围） | P1 |
| 4 | 入口配置 API | 创建/修改资源位入口（标题、图片、链接、时段、客户范围） | P1 |
| 5 | maps 落地页 API | 创建落地页、上传头图、挂载优惠券、导入商品 | P1 |
| 6 | 审批流 API | 提交审批单到内部系统 + 查询/回调审批结果 | P1 |
| 7 | 商品查询 API | 按品牌/品类查询商品列表 | P1 |
| 8 | 图片上传 API | 上传封面图/头图到后台 | P2 |
| 9 | 企微消息推送 API | 推送日报/设计需求单/审批结果到企微群 | P2 |

### 关于腾讯文档 API

腾讯文档开放平台支持通过 API 读写表格数据，技术团队封装一次即可：
- 读取排期表 → 指定 sheet + 日期范围，返回各资源位占用情况
- 写入排期 → 指定资源位 + 日期 + 品牌，更新对应单元格
- 不需要迁移排期表，运营和业务方继续在腾讯文档里看，AI 在背后读写同一份表

## 六、产品支持模式的转变

### 传统模式

```
业务方有推广需求
  → 找平台运营沟通
  → 平台运营逐个执行 15 个环节
  → 每次都是重复劳动
```

### OpenClaw 模式

```
产品团队：搭建 1 个"资源位全流程管理"标杆 Skill，跑通一次完整配置
技术团队：封装 9 个底座 API
运营团队：
  ├── 可以自己做的：调整推荐规则、修改数据播报模板、更新话术
  └── 做不了的：新增资源位类型的 API 对接 → 提给产品和技术
```

### 三个角色的分工

| 角色 | 职责 | 频率 |
|------|------|------|
| 产品团队 | 搭建标杆 Skill，定义全流程规范 | 一次性 |
| 技术团队 | 封装底座 API（腾讯文档、神策、后台配置、maps、审批流） | 按需 |
| 运营团队 | 使用 Skill 完成日常资源位管理，调整规则和参数 | 高频、自主 |

## 七、落地节奏

### Phase 1（第 1-3 周）—— 数据播报 + 截图识别推荐

**目标**：先跑通数据能力和需求识别，不改变现有配置流程。

| 工作项 | 负责方 |
|-------|-------|
| 封装神策数据查询 API | 技术 |
| 封装腾讯文档读取 API | 技术 |
| 搭建"数据日报播报"模块 | 产品 |
| 搭建"截图识别+排期推荐"模块 | 产品 |

**交付成果**：
- 每日自动生成资源位数据日报，推送到企微群
- 截图业务方消息 → AI 自动推荐资源位 + 生成回复话术

**价值**：运营每天省 30min 拉数据，对接业务方效率翻倍。

### Phase 2（第 4-6 周）—— 自动配置 + 审批上线

**目标**：打通配置到上线的自动化链路。

| 工作项 | 负责方 |
|-------|-------|
| 封装优惠券创建 API | 技术 |
| 封装入口配置 API | 技术 |
| 封装 maps 落地页 API | 技术 |
| 封装审批流 API | 技术 |
| 封装腾讯文档写入 API | 技术 |
| 搭建"配置执行"模块 | 产品 |
| 搭建"审批上线"模块 | 产品 |

**交付成果**：
- 运营确认后，自动完成排期锁定、优惠券创建、落地页搭建、入口配置
- 自动提交审批，审批通过自动上线

**价值**：配置环节从 2-3 小时缩短到 3 次确认。

### Phase 3（第 7-9 周）—— 全链路串联 + 优化闭环

**目标**：串联全流程，补齐辅助能力。

| 工作项 | 负责方 |
|-------|-------|
| 封装商品查询 API | 技术 |
| 封装企微推送 API | 技术 |
| 搭建"商品推荐"能力 | 产品 |
| 搭建"设计需求单自动生成" | 产品 |
| 搭建"效果分析+优化建议" | 产品 |

**交付成果**：
- 一张截图触发完整链路：需求识别 → 推荐 → 排期 → 设计需求 → 配置 → 审批 → 上线 → 监控
- 数据驱动的优化建议（换素材、调时段、调位置）

**价值**：平台运营从"执行者"转型为"决策者"。

## 八、预期效果

| 指标 | 当前 | Phase 1 后 | 全部落地后 |
|------|------|-----------|----------|
| 单次资源位配置耗时 | 2-3 小时 | 1.5-2 小时 | 15 分钟（3 次确认） |
| 数据监控频率 | 偶尔手动拉 | 每日自动播报 | 每日播报+实时预警 |
| 效果优化建议 | 基本没做 | 异常提醒 | 自动分析+优化建议 |
| 运营可支撑的资源位数量 | 受人力限制 | 小幅提升 | 大幅提升（执行全自动） |
| 业务方等待响应时间 | 半天到一天 | 10 分钟内 | 即时响应 |
