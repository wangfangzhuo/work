# OpenClaw 落地场景：资源位全流程智能化

## 一、背景与痛点

### 平台运营的核心工作

平台运营日常需要对接 MP 商家运营、JBP 商家运营、厂牌精选运营、销售运营等多个团队，负责平台资源位的排期规划、配置上线和数据监控。

### 资源位类型

平台资源位覆盖 PC 端和 APP 端，类型繁多：

**PC 端**（首页设置）：首页轮播图、顶通广告、搜索热词、商品楼层、公告区、药城精选(商品)、药城精选(品牌/厂家)、导航TAB、质量公示等

**APP 端**（首页配置）：轮播图、背景图、信息流区域、信息流广告、开屏广告、导航按钮、秒杀专区、中通广告、药城精选、品牌推荐、系统推荐、搜索发现、新一起系列、药城公告、侧边栏广告、弹窗、系统推荐V3等

### 一次完整的资源位配置 = 16 个环节

```
① 业务方提需求（企微沟通）
    ↓
② 查排期、推荐资源位
    ↓
③ 确认排期，锁定资源位
    ↓
④ 沟通活动细节（主题、优惠券、商品、客户范围）
    ↓
⑤ 创建用户集（如需精细圈选用户）
    ↓
⑥ 创建优惠券（关联用户集）
    ↓
⑦ 提设计需求（给设计师）
    ↓
⑧ 设计师出图
    ↓
⑨ 收集活动商品
    ↓
⑩ 配置落地页（maps：头图 + 优惠券 + 导入商品）
    ↓
⑪ 配置入口（标题、封面图、链接、时段、客户范围）
    ↓
⑫ 上线前检查
    ↓
⑬ 提交审批（内部系统审批通过后才可上线）
    ↓
⑭ 上线
    ↓
⑮ 数据监控
    ↓
⑯ 效果反馈与优化建议
```

### 核心矛盾

- 单个环节都不复杂，但 **16 个环节串起来，涉及 4 个角色（业务方、平台运营、设计师、采购）、5 个系统（腾讯文档、CRM、后台配置、maps、神策）**
- 平台运营 70%+ 时间花在执行操作上，没有精力做策略优化
- 每周四沟通下周排期，涉及多个团队的需求汇总与冲突协调
- 数据效果监控靠手动拉取，基本没有后续优化动作

- 每周四，平台运营找各业务方沟通确定下一周排期
- 业务方优先选有排期的资源位里流量最大的
- 最大的被占了但又想用 → 拉群聊沟通优先级

## 三、用户范围与优惠券配置详情

### 用户范围选定：两种模式

| 模式 | 适用场景 | 操作方式 | 涉及系统 |
|------|---------|---------|---------|
| **简单模式** | 按客户省份/客户类型投放 | 直接在券/资源位配置里选择 | 后台配置 |
| **精细模式** | 按用户行为数据圈选或指定用户ID | 先在 CRM 创建用户集 → 再关联到券/资源位 | CRM + 后台配置 |

### CRM 用户集创建

用户集支持多维度条件组合（且/或关系）：

| 维度类型 | 可选条件 |
|---------|--------|
| **客户属性** | 客户省份、客户类型、企业名称等基础属性 |
| **业务快照** | 最后查看购物车时间、最后点击去结算时间、最后下单时间、最后支付时间、最后拜访时间、周期内加车次数、周期内点击去结算次数、周期内订单数、周期内GMV、周期内订单出库GMV |
| **贡献排名** | 按贡献度排名筛选 |
| **关联商品** | 买过/未买过某些商品的用户 |
| **关联已有集合** | 复用已创建的用户集 |
| **用户分层** | 用户价值分层 |
| **本地属性** | 本地化属性筛选 |

用户集配置项：
- 分组方式：自动计算 / 固定不变 / 任务
- 是否资源位：是 / 否
- 时间条件：动态时间段（X天以内/以外）/ 固定时间段（日期范围）/ 按周期

### 优惠券创建完整字段

| 字段 | 说明 |
|------|------|
| 优惠券名称 | 文本 |

| 有效期 | 绝对时间（日期范围）/ 相对时间 / 领取即生效 |

| 适用商家 | 全平台 / 部分店铺（可选择店铺、选择区域、加载MP交接、导入店铺） |
| 适用商品 | 全部商品 / 部分商品 / 剔除商品（支持单个添加或批量导入） |

## 四、OpenClaw 改造方案

### 改造前 vs 改造后

| 维度 | 当前 | OpenClaw 之后 |
|------|------|--------------|
| 一次配置耗时 | 2-3 小时（分散在多天） | 3 次确认，执行全自动 |
| 运营角色 | 执行者（逐个操作 16 个环节） | 决策者（确认排期、确认商品、确认上线） |
| 数据监控 | 手动拉、偶尔做 | 每日自动播报 + 异常预警 + 优化建议 |
| 效果反馈 | 基本没做 | 自动生成效果报告推送给业务方 |

### 理想流程演示

```
平台运营截图业务方的企微消息丢给 AI
  ↓
[Skill] "我识别到以下需求：
  品牌：九芝堂
  期望时间：下周二至周四
  偏好：首页、流量大

  查询排期后推荐：
  1. 首页中通2（周二至周四全空，日均点击 XXX）✅ 推荐
  2. 首页弹窗（周三周四空闲，日均点击 XXX）
  3. 首页中通1（已被本地省心购占用）❌

  是否锁定首页中通2？"
  ↓
运营："锁定，活动是满199减30，面向30天内下过单的单体药店，
      商品用九芝堂的感冒灵系列"
  ↓
[Skill] 自动执行：
  ✅ 更新腾讯文档排期表 → 首页中通2 周二至周四 = 九芝堂
  ✅ 创建用户集 → CRM 圈选条件：客户类型=单体药店 且 最后下单时间=30天以内
     → 命中客户 3,267 家
  ✅ 创建优惠券 → 满199减30，关联用户集，适用商品=九芝堂感冒灵系列
  ✅ 生成设计需求单 → 推送给设计师
     "九芝堂品牌推广，首页中通2入口图（750x360px）
      + 落地页头图（750x420px），周一下班前交付"
  ✅ 推荐商品列表 → 九芝堂感冒灵相关 12 个 SKU，请确认
  ↓
运营确认商品列表 ✅，设计师交付图片 ✅
  ↓
运营："图片好了，商品确认，上线吧"
  ↓
[Skill] 自动执行：
  ✅ 创建落地页（maps）→ 头图 + 优惠券 + 导入商品
  ✅ 配置入口 → 标题、封面图、链接、时段、企业类型=单体药店
     生效周期=周二至周四
  ✅ 上线前校验 → 图片、链接、时段、商品库存全部通过
  ✅ 提交审批 → 推送到内部审批系统，附配置摘要和预览：
     "【资源位上线审批】
      资源位：首页中通2
      品牌：九芝堂
      时段：3/25（周二）至 3/27（周四）
      活动：满199减30
      用户集：30天内下单的单体药店（3,267家）
      商品：12个SKU
      落地页预览：[链接]
      入口封面预览：[图片]"
  ↓
  ⏸ 等待审批（人工审批）
  ↓
  审批通过 → [Skill] 自动发布上线
  审批驳回 → [Skill] 通知运营驳回原因，提示修改项
  ↓
[Skill] 上线后每日自动：
  ✅ 拉取神策数据 → 生成日报
  ✅ 异常预警 → 点击率低于均值自动提醒
  ✅ 活动结束 → 自动生成效果总结推送给业务方
```

**平台运营从"执行 16 个环节"变成"确认 3 次 + 1 次审批"。**

## 五、16 个环节逐一拆解

| # | 环节 | 当前做法 | OpenClaw 之后 | 自动化程度 |
|---|------|--------|-------------|:---------:|
| ① | 业务方提需求 | 企微群聊沟通 | 截图识别 → 自动提取品牌、时间、偏好等结构化信息 | 全自动 |
| ② | 查排期+推荐资源位 | 看腾讯文档排期表，凭经验推荐 | 自动读取腾讯文档排期表 + 按历史流量排序推荐 + 生成回复话术 | 全自动 |
| ③ | 锁定排期 | 手动更新腾讯文档 | 运营确认后自动更新腾讯文档排期表 | 全自动 |
| ④ | 沟通活动细节 | 企微反复沟通 | AI 生成确认清单（主题？券面额？商品范围？用户范围？），运营一次性发给业务方确认 | AI 辅助 |
| ⑤ | 创建用户集 | CRM 后台手动创建条件组合 | 运营描述圈选条件，自动调 CRM API 创建用户集，返回命中客户数供确认 | 全自动 |
| ⑥ | 创建优惠券 | 后台手动填写十余个字段 | 运营描述券规则，自动调 API 创建并关联用户集 | 全自动 |
| ⑦ | 提设计需求 | 在线表格提需求给设计师 | 基于需求信息自动生成设计需求单（品牌、主题、尺寸、交付时间），推送给设计师 | 全自动 |
| ⑧ | 设计师出图 | 设计师排期制作 | AI 不介入，等待设计师交付 | 人工 |
| ⑨ | 收集活动商品 | 业务方提供 Excel 或运营整理 | AI 按品牌+活动类型自动推荐商品列表，确认后导出 | AI 辅助 |
| ⑩ | 配置落地页 | maps 手动搭建（头图+券+商品） | 自动调 maps API：上传头图、挂载优惠券、导入商品 | 全自动 |
| ⑪ | 配置入口 | 后台逐个填字段 | 自动调后台 API 填充所有字段（标题、图片、链接、时段、生效周期、企业类型、客户标签） | 全自动 |
| ⑫ | 上线前检查 | 人工逐项核查 | 自动校验：图片是否上传、链接是否可访问、时段是否正确、商品是否有库存 | 全自动 |
| ⑬ | 提交审批 | 人工提交 | 自动提交审批单到内部系统，附配置摘要+预览链接 | 全自动 |
| ⑭ | 上线 | 审批通过后手动发布 | 审批通过自动发布；驳回自动通知运营并提示修改项 | 全自动 |
| ⑮ | 数据监控 | 手动从神策拉数据 | 每日自动拉取，生成日报+异常预警 | 全自动 |
| ⑯ | 效果反馈 | 基本没做 | 自动生成效果报告+优化建议，推送给业务方 | 全自动 |

**16 个环节中：12 个全自动，2 个 AI 辅助，2 个人工（设计出图 + 审批）。**

## 六、Skill 设计

### 一个 Skill，内部 6 个模块

```
资源位全流程管理 Skill
├── 模块1：需求识别（截图识别 → 结构化提取）
├── 模块2：排期推荐（读取腾讯文档排期表 → 流量排序 → 推荐+话术）
├── 模块3：用户圈选（解析圈选条件 → 调 CRM API 创建用户集）
├── 模块4：配置执行（优惠券+落地页+入口 → 一键配置）
├── 模块5：审批上线（自动提交审批 → 监听结果 → 发布/通知）
└── 模块6：数据监控（日报播报 → 异常预警 → 效果总结）
```

## 七、需要技术封装的底座 API

| # | API | 说明 | 优先级 |
|---|-----|------|:-----:|
| 1 | 腾讯文档读写 API | 读取/更新排期表（通过腾讯文档开放 API，OAuth 授权） | P0 |
| 2 | 神策数据查询 API | 按资源位+时间维度查询点击、访问、转化数据 | P0 |
| 3 | CRM 用户集创建 API | 按条件组合创建用户集（支持客户属性、业务快照、关联商品等维度，且/或关系） | P1 |
| 4 | CRM 用户集查询 API | 查询已有用户集列表、客户数、条件详情 | P1 |
| 5 | 优惠券创建 API | 创建优惠券（名称、面额、门槛、有效期、适用用户/商家/商品、限领规则、预算）并关联用户集 | P1 |
| 6 | 入口配置 API | 创建/修改资源位入口（标题、图片、链接、时段、生效周期、企业类型、客户标签） | P1 |
| 7 | maps 落地页 API | 创建落地页、上传头图、挂载优惠券、导入商品 | P1 |
| 8 | 审批流 API | 提交审批单到内部系统 + 查询/回调审批结果 | P1 |
| 9 | 商品查询 API | 按品牌/品类查询商品列表 | P1 |
| 10 | 图片上传 API | 上传封面图/头图到后台 | P2 |
| 11 | 企微消息推送 API | 推送日报/设计需求单/审批结果到企微群 | P2 |

### 关于腾讯文档 API

腾讯文档开放平台支持通过 API 读写表格数据，技术团队封装一次即可：
- 读取排期表 → 指定 sheet + 日期范围，返回各资源位占用情况
- 写入排期 → 指定资源位 + 日期 + 品牌，更新对应单元格
- 不需要迁移排期表，运营和业务方继续在腾讯文档里看，AI 在背后读写同一份表
- 工作量预估：1-2 天

## 八、产品支持模式的转变

### 传统模式

```
业务方有推广需求
  → 找平台运营沟通
  → 平台运营逐个执行 16 个环节
  → 每次都是重复劳动
```

### OpenClaw 模式

```
产品团队：搭建 1 个"资源位全流程管理"标杆 Skill，跑通一次完整配置
技术团队：封装 11 个底座 API
运营团队：
  ├── 可以自己做的：调整推荐规则、修改数据播报模板、更新话术、调整用户圈选条件
  └── 做不了的：新增资源位类型的 API 对接 → 提给产品和技术
```

### 三个角色的分工

| 角色 | 职责 | 频率 |
|------|------|------|
| 产品团队 | 搭建标杆 Skill，定义全流程规范 | 一次性（+ 偶尔新场景） |
| 技术团队 | 封装底座 API（腾讯文档、CRM、神策、后台配置、maps、审批流） | 按需 |
| 运营团队 | 使用 Skill 完成日常资源位管理，调整规则和参数 | 高频、自主 |

**一句话：产品从"帮运营配置每一个资源位"变成"帮运营搭建自己管理资源位的能力"。**

## 九、落地节奏

### Phase 1（第 1-3 周）—— 数据播报 + 截图识别推荐

**目标**：先跑通数据能力和需求识别，不改变现有配置流程。

| 工作项 | 负责方 |
|-------|-------|
| 封装神策数据查询 API | 技术 |
| 封装腾讯文档读取 API | 技术 |
| 搭建"数据日报播报"模块 | 产品 |

### Phase 2（第 4-6 周）—— 用户圈选 + 自动配置 + 审批上线

**目标**：打通从用户圈选到审批上线的自动化链路。

| 工作项 | 负责方 |

| 封装审批流 API | 技术 |
| 封装腾讯文档写入 API | 技术 |
| 搭建"用户圈选"模块 | 产品 |
| 搭建"配置执行"模块 | 产品 |

**交付成果**：

- 运营确认后，自动完成排期锁定、落地页搭建、入口配置
- 自动提交审批，审批通过自动上线

**价值**：配置环节从 2-3 小时缩短到 3 次确认 + 1 次审批。
