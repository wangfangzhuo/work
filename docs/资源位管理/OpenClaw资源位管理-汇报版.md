## P1｜我理解的 OpenClaw：感知-记忆-执行-反馈（30s）

OpenClaw 的本质是**给每个业务场景配一个 AI 运营助理**，它的能力模型可以抽象为四层：

```
┌─────────────────────────────────────────┐
│           感知层（Perceive）              │
│   理解非结构化输入 → 提取结构化意图        │
├─────────────────────────────────────────┤
│           记忆层（Memory）               │
│   业务知识库 + 历史数据 → 经验沉淀        │
├─────────────────────────────────────────┤
│           执行层（Act）                  │
│   跨系统 API 编排 → 一次指令多步完成      │
├─────────────────────────────────────────┤
│           反馈层（Feedback）              │
│   数据监控 + 异常预警 + 效果归因          │
└─────────────────────────────────────────┘
```

| 能力层 | 通用定义 | 本次场景举例 |
|-------|---------|-----------|
| **感知** | 从非结构化输入中提取业务意图 | 截图企微聊天 → 识别品牌、时间、偏好 |
| **记忆** | 沉淀业务规则和历史数据为可调用的知识 | 资源位流量基线、排期状态、团队偏好 |
| **执行** | 跨多个系统完成端到端操作 | 一句话跨 5 个系统完成配置 |
| **反馈** | 主动监控、预警、归因 | 每日播报数据，异常自动告警 |

**这套四层模型不只适用于资源位，任何"多系统、重执行、规则明确"的运营场景都可以复用。**

## P2｜痛点：94% 的时间在执行（60s）

**一次资源位配置 = 16 个环节 × 4 个角色 × 5 个系统**

```
运营一天 8 小时：

██████████████████░░  3.0h  搬运数据和配置     ← AI 可替代
████████░░░░░░░░░░░░  2.0h  群里沟通对齐需求   ← AI 可辅助
██████░░░░░░░░░░░░░░  1.5h  拉数据做报表       ← AI 可替代
████░░░░░░░░░░░░░░░░  1.0h  审批和检查         ← AI 可替代
██░░░░░░░░░░░░░░░░░░  0.5h  策略思考和优化     ← 这才是运营该做的事
```

**94% 执行，6% 思考。OpenClaw 要翻转这个比例。**

## P3｜16 个环节逐一拆解（60s）

| # | 环节 | 当前做法 | OpenClaw 之后 | 自动化 |
|---|------|--------|-------------|:-----:|
| ① | 业务方提需求 | 企微群聊沟通 | 截图识别 → 自动提取品牌、时间、偏好 | 全自动 |
| ② | 查排期+推荐资源位 | 看腾讯文档，凭经验推荐 | 自动读取排期表 + 按流量排序推荐 + 生成回复话术 | 全自动 |
| ③ | 锁定排期 | 手动更新腾讯文档 | 运营确认后自动更新排期表 | 全自动 |
| ④ | 沟通活动细节 | 企微反复沟通 | AI 生成确认清单，运营一次性发给业务方确认 | AI辅助 |
| ⑤ | 创建用户集 | CRM 手动创建条件组合 | 运营描述圈选条件，自动调 CRM API 创建 | 全自动 |
| ⑥ | 创建优惠券 | 后台手动填十余个字段 | 运营描述券规则，自动调 API 创建并关联用户集 | 全自动 |
| ⑦ | 提设计需求 | 在线表格提需求 | 自动生成设计需求单（品牌、主题、尺寸、交付时间） | 全自动 |
| ⑧ | 设计师出图 | 设计师排期制作 | 短期人工（AI 生图能力已具备，但品牌视觉需求频繁调整，暂由设计师把控） | 人工* |
| ⑨ | 收集活动商品 | 业务方提供 Excel | 自动读取企微/腾讯文档中的商品清单（智能选品可作为独立能力单独抽象，本场景暂不细化） | 全自动 |
| ⑩ | 配置落地页 | maps 手动搭建 | 自动调 maps API：上传头图、挂载券、导入商品 | 全自动 |
| ⑪ | 配置入口 | 后台逐个填字段 | 自动调 API 填充所有字段 | 全自动 |
| ⑫ | 上线前检查 | 人工逐项核查 | 自动校验图片、链接、时段、库存 | 全自动 |
| ⑬ | 提交审批 | 人工提交 | 自动提交审批单，附配置摘要+预览链接 | 全自动 |
| ⑭ | 上线 | 审批通过后手动发布 | 审批通过自动发布；驳回自动通知并提示修改项 | 全自动 |
| ⑮ | 数据监控 | 手动从神策拉数据 | 每日自动拉取，生成日报+异常预警 | 全自动 |
| ⑯ | 效果反馈 | 基本没做 | 自动生成效果报告+优化建议，推送给业务方 | 全自动 |

**16 个环节：13 个全自动，1 个 AI 辅助，1 个人工*，1 个独立能力待抽象。**

> *⑧ 设计出图：AI 生图技术已成熟，但品牌方视觉需求变化快，短期先由设计师把控质量，中长期可逐步 AI 化。

## P5｜Phase 1 实现方案：1 个 Skill + 3 个底座 API（60s）

**先做最小闭环：假设素材齐了，Agent 只做"配置入口→检查→审批→上线"。**

### Skill 架构（Phase 1）

```
资源位入口配置 Skill
├── 模块1：入口配置（解析运营指令 → 调 API 填充所有字段）
├── 模块2：上线前检查（图片尺寸、链接可访问、时段无冲突、字段完整性）
└── 模块3：审批上线（自动提交审批单 → 监听结果 → 发布/通知）
```

### 需要技术封装的底座 API（Phase 1）

| # | API | 说明 | 工作量预估 |
|---|-----|------|:--------:|
| 1 | 入口配置 API | 创建/修改资源位入口（标题、图片、链接、时段、生效周期、企业类型） | 2-3 天 |
| 2 | 图片上传 API | 上传封面图/头图到后台，返回图片 URL | 1 天 |
| 3 | 审批流 API | 提交审批单到内部系统 + 查询/回调审批结果 | 2-3 天 |

### 后续阶段扩展的 API

| 阶段 | API | 说明 |
|:----:|-----|------|
| Phase 2 | CRM 用户集创建/查询 API | 按条件组合创建用户集 |
| Phase 2 | 优惠券创建 API | 创建优惠券并关联用户集 |
| Phase 2 | maps 落地页 API | 创建落地页、上传头图、挂载券、导入商品 |
| Phase 3 | 腾讯文档读写 API | 读取/更新排期表 |
| Phase 3 | 神策数据查询 API | 按资源位+时间维度查询数据 |
| Phase 3 | 商品查询 API | 按品牌/品类查询商品列表 |
| Phase 3 | 企微消息推送 API | 推送日报/审批结果到企微群 |

```
Phase 1                    Phase 2                      Phase 3

┌──────────┐          ┌──────────────┐          ┌─────────────────┐

│ 已有落地页 │    →     │ + 优惠券创建   │    →     │ + 排期推荐        │
│ 已有用户集 │          │ + 落地页配置   │          │ + 数据监控        │
│          │          │              │          │ + 效果反馈        │
│ Agent 配置 │          │ Agent 配置    │          │ + 截图触发全链路   │
│ 资源位入口 │          │ 资源位入口     │          │                 │
└──────────┘          └──────────────┘          └─────────────────┘
  覆盖⑪⑫⑬⑭             覆盖⑤⑥⑩⑪⑫⑬⑭           覆盖全部16个环节
```

| 阶段 | 一句话 | 前提假设 | 覆盖环节 | 交付成果 |
|------|-------|---------|---------|---------|
| **Phase 1：最小闭环** | 素材齐了，Agent 帮你一键配置上线 | maps 落地页链接已有、主图已有、用户集 ID 已有 | ⑪配置入口 → ⑫检查 → ⑬审批 → ⑭上线 | 入口配置从 30min 降至 1min |

## P8｜产品经理的价值：不是成本，是生产"成本节省"的人（90s）

**PM 不是被 AI 替代的人，PM 是拿 AI 去替代重复劳动的人。**

### 1. 定义"做什么"和"不做什么"

AI 能跑完 16 个环节，但**哪些该自动化、哪些必须留给人**，是 PM 的判断：
- 审批不能自动通过——合规红线不可逾越
- 设计出图不用 AI——品牌方对视觉有主观要求
- 这些边界判断，AI 没有立场，PM 有

### 2. 把散落的经验变成可执行的规则

运营的经验在脑子里、群聊里、Excel 里。**谁来定义知识库结构、谁来决定什么该沉淀什么该丢弃**，是 PM 的工作：
- "化学药可以勾选生化药品和抗生素制剂"——怎么变成 Skill 逻辑？
- "流量大的位置优先给 JBP 商家"——要不要写进推荐算法？
- AI 是执行规则的，PM 是**制定规则的**

### 3. 设计人和 AI 的协作边界

**什么时候让 AI 自己跑、什么时候必须停下来等人确认**，这个流程设计决定了系统的可靠性：
- 用户集创建完要让运营确认命中人数——防止圈选条件写错影响几千家客户
- 优惠券预算要设上限拦截——防止配置错误造成资损
- 这些"安全阀"的位置，是 PM 设计的

### 4. 跨团队的需求翻译和优先级决策

AI 不会参加排期会，不会判断"这周九芝堂和北京同仁堂谁优先级更高"：
- 业务价值的权衡、团队间利益的协调、资源冲突的裁决——这些是人的判断
- PM 是**业务和技术之间的翻译官**：把运营痛点翻译成 Skill，把技术边界翻译成产品方案

### 5. 持续量产 AI 能力，ROI 3-5 倍

```
1 个 PM 设计 1 个 Skill → 资源位场景省 80 万/年
做完资源位，接着做资质审核、商品对码、客户注册……
每多做一个场景，就多省一份成本

PM 的 ROI = 产出的降本金额 ÷ PM 自身成本 ≈ 3-5 倍
```

| 场景 | 预估节省 | 状态 |
|------|---------|------|
| 资源位全流程智能化 | 80 万/年 | ✅ 本次方案 |
| 资质审核自动化 | 待评估 | 下一个 Skill |
| 商品对码智能化 | 待评估 | 下一个 Skill |
| 客户注册自助化 | 待评估 | 下一个 Skill |

**每个 PM 就是一台"AI 能力生产机"——一个场景接一个场景，持续为公司量产降本增效的能力。**

### 一句话总结

> **AI 时代，PM 的价值不是"会画原型"，而是"能把业务问题变成 AI 方案，持续为公司量产降本增效的能力"。这份方案本身，就是最好的证明。**
