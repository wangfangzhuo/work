## Slide 1 — 封面

**OpenClaw 分享会**
**平台运营场景与思考**

不是把文档内容搬上来，而是回答两个问题：
1. 平台运营最值得做什么
2. 怎么从"人工操作"走到"规则驱动的自动化运营"

平台运营三大核心职责：
- **活动发布与配置**：发布平台活动，配合 JBP/MP/采购等团队进行商品提报和配置维护
- **资源位管理**：规划资源位排期，跨团队协商需求，完成配置（类型多、需求大、协作复杂）
- **数据追踪与行动建议**：追踪日常数据和活动数据，评估各项目效果，给出后续行动项

## Slide 3 — 为什么平台运营值得优先做

**运营的三大职责天然具备 Skill 化条件**

| 核心职责 | 当前痛点 | 为什么适合 Skill 化 |
|---------|---------|-------------------|
| **活动发布与配置** | 每次活动都要手动提报、逐条审批、逐个配置满减，配合多团队（JBP/MP/采购）反复沟通 | 规则明确、动作重复、系统可连，是最典型的 SOP 自动化场景 |
| **资源位管理** | 资源位类型多、需求方多（营销/运营/商家/采购），排期靠 Excel + 群里协商，冲突频发 | 排期是规则+约束的调度问题，天然适合 AI 辅助决策 |
| **数据追踪与行动建议** | 手动拉数据、做表格、写分析，耗时长且对人的分析能力依赖大 | 数据结构化、分析逻辑可沉淀，最适合 AI 做"初稿"人做"决策" |

**核心矛盾：**
运营团队 70%+ 时间花在配置、协调、搬数据上，真正该做的策略思考和规则优化反而没时间做。

## Slide 4 — 平台运营全链路地图

**围绕三大职责，画出完整的工作链路**

```
一、活动发布与配置
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
活动策划 → 跨团队沟通（JBP/MP/采购）→ 商品提报 → 提报审批 → 满减/券配置 → 上线检查
   ↑                                                                    │
   └──────────────────── 复盘数据反哺下次策划 ←──────────────────────────────┘

二、资源位管理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
需求收集（多团队）→ 排期规划 → 冲突协调 → 资源位配置 → 上线 → 效果追踪
                      ↕
            各团队需求变更、临时插入

三、数据追踪与行动建议
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
日常数据监控 → 活动数据追踪 → 效果归因分析 → 输出行动建议 → 推动执行
```

**思考重点：**
三条线不是孤立的——活动配置产生数据，数据分析反哺下次活动策略，资源位排期影响活动效果。
OpenClaw 的价值是打通这三条线，让数据驱动决策、让规则驱动执行。

## Slide 6 — 重点场景 A：活动发布与配置自动化

**这是运营每天最耗时的工作，也是最容易做出"看得见效率"的场景**

### 当前工作流 vs OpenClaw 之后

| 环节 | 当前（人工） | OpenClaw 之后 |
|------|------------|--------------|
| 商品筛选 | 运营手动导数据、Excel 筛选符合条件的商品 | Skill 按活动规则自动筛选，输出候选清单 |
| 跨团队沟通 | 群里 @JBP/MP/采购，反复确认提报商品 | Skill 自动汇总各团队提报，冲突自动标注 |
| 提报审批 | 人工逐条审核，驳回手写理由 | 合规的秒批，驳回自动附整改建议发给对应团队 |
| 满减配置 | 手动在后台逐个商品配置满减/券/打标 | Skill 调用 API 批量完成配置 |
| 上线检查 | 人工逐项核查配置是否正确 | Skill 自动校验配置一致性，异常自动标红 |

### 一个大促场景的完整闭环

> **38 节大促：**
> 1. 运营设定活动规则（门槛、满减机制、目标 GMV）
> 2. Skill 自动筛选符合条件的商品，推送给 JBP/MP/采购团队确认
> 3. 各团队提报后，Skill 自动审批合规商品，驳回的自动发整改建议
> 4. 审批通过的商品，Skill 调用后台 API 完成满减配置与打标
> 5. 上线前 Skill 自动校验全部配置，异常项推送给运营确认
>
> **自主汇报：** "38 节商品提报完成：收到 JBP 提报 800 个、MP 提报 1200 个、采购提报 500 个，自动审批通过 2100 个，驳回 400 个（已发整改建议），配置已全部完成，3 个配置异常已标注待确认。"

## Slide 7 — 重点场景 B：资源位智能排期

**资源位管理是运营最头疼的协调工作——类型多、需求方多、冲突频发**

### 当前痛点

- 资源位类型繁多（首页轮播、类目入口、搜索推荐、弹窗、频道页等）
- 需求方来自公司各团队（营销、JBP 运营、MP 运营、采购、品牌合作等）
- 排期靠 Excel 管理 + 群里沟通协商，冲突后反复调整
- 临时插入需求频繁，打乱已有排期

### OpenClaw 能做什么

```
各团队提交资源位需求（时间、位置、优先级、素材）
    ↓
[Skill] 自动汇总所有需求，识别时间/位置冲突
    ↓
[Skill] 基于优先级规则 + 历史效果数据，推荐排期方案
    ├── 无冲突 → 直接输出排期表
    └── 有冲突 → 列出冲突项 + 建议调整方案，推送给相关团队协商
    ↓
运营确认后，Skill 自动完成资源位内容配置
    ↓
上线后自动追踪各资源位点击率/转化率
```

### 价值

- **排期效率**：从"群里反复沟通 2-3 天"降到"Skill 出方案 + 确认 2 小时"
- **冲突减少**：规则化的优先级判断，减少人为协调摩擦
- **效果可追溯**：每个资源位的效果数据自动关联，下次排期有据可依

## Slide 8 — 重点场景 C：数据追踪与行动建议

**运营不缺数据，缺的是"从数据到行动"的最后一公里**

### 三个层次

| 层次 | 能力 | 示例 |
|------|------|------|
| **L1 数据播报** | 每日/每周自动推送核心数据摘要 | "今日 GMV 较昨日 -8%，主要下降品类：感冒类 OTC（-15%）" |
| **L2 效果归因** | 活动结束后自动分析效果，对比历史同类活动 | "38 节 vs 去年 38 节：GMV +12%，但满减核销率下降 5%，原因：门槛偏高" |
| **L3 行动建议** | 基于分析结果，自动输出可执行的行动项 | "建议：① 下次满减门槛从 200 降到 150；② 感冒类增加 3 个资源位曝光；③ 召回上次未参与的 TOP50 商家" |

### 当前 vs OpenClaw 之后

| 环节 | 当前（人工） | OpenClaw 之后 |
|------|------------|--------------|
| 数据获取 | 手动登录多个后台，导出 Excel | Skill 自动拉取，定时推送摘要 |
| 数据分析 | 运营手动做透视表、写分析 | Skill 自动归因，输出结构化报告 |
| 行动建议 | 依赖个人经验，质量参差不齐 | Skill 基于历史数据 + 规则给出建议，人做决策 |
| 推动执行 | 群里 @相关人，手动追踪进度 | Skill 自动将行动项推送给责任人，追踪完成状态 |

### 自主汇报示例

> "本周数据周报：整体 GMV 环比 +5%，其中百亿补贴贡献 +8%。3 个需要关注的点：① 感冒类复购率连续 2 周下降，建议排查价格竞争力；② 资源位 A3 点击率仅 1.2%，建议更换素材；③ JBP 商家 TOP10 中有 3 家本周零动销，建议客户经理跟进。行动项已推送给对应负责人。"

## Slide 9 — 怎么排优先级

**不是谁最酷，而是谁最先跑出结果**

用"业务价值 × 数据可用 × 接入难度 × 闭环潜力"来排：

```
                        业务价值 / 闭环潜力 →
                   ┌──────────────────────────────────┐
                   │                                  │
  数据可用         │  日常数据播报     活动配置自动化 ★   │  高价值/易落地
  接入容易         │  跨团队协作助手   活动效果复盘       │  → 优先开做
       ↓          │                                  │
                   ├──────────────────────────────────┤
                   │                                  │
                   │  资源位配置      资源位智能排期     │  高价值/难落地
                   │                 行动建议生成       │  → 作为二期
                   │                                  │
                   └──────────────────────────────────┘
```

**第一批建议：**

| 定位 | 场景 | 理由 |
|------|------|------|
| **标杆场景** | 活动配置自动化 | 运营每天最耗时的工作、规则明确、系统可连 |
| **快速见效** | 日常数据播报 | 不依赖复杂系统对接，有数据就能做 |
| **效率提升** | 活动效果复盘 | 每次活动都需要，手动做最痛 |

## Slide 10 — 落地方法论：从"问答助手"到"运营闭环"

**分 4 个层级推进，每升一层都更有价值**

```
L1 解答          →    L2 诊断           →    L3 代办            →    L4 闭环
━━━━━━━━              ━━━━━━━━               ━━━━━━━━                ━━━━━━━━
能回答活动规则         能分析活动效果          能自动配置满减           全自动活动周期
能查询资源位排期       能识别数据异常          能预填提报表单           全自动资源位排期
能解释操作流程         能推荐排期方案          能自动生成复盘报告        全自动数据→行动闭环
```

### 演进路径

| 阶段 | 活动配置 | 资源位管理 | 数据追踪 | 预计时间 |
|------|---------|----------|---------|---------|
| **Phase 1** | 回答活动规则和操作指引 | 查询排期、展示冲突 | 自动播报日常数据 | 0-3 个月 |
| **Phase 2** | 自动筛选商品、推荐提报 | 推荐排期方案 | 自动归因分析 | 3-6 个月 |
| **Phase 3** | 自动审批+配置+校验 | 自动配置资源位 | 自动输出行动建议 | 6-9 个月 |
| **终态** | 全自动活动闭环 | 全自动排期+配置+追踪 | 数据→分析→行动→追踪 | 9-12 个月 |

## Slide 11 — 组织打法：不要让产品团队一个个写死场景

**更好的方式，是产品做模板、运营来定义规则、平台做保障**

```
产品团队                    运营团队                      平台/技术
━━━━━━━━━                  ━━━━━━━━━                    ━━━━━━━━━
· 做 1-2 个标杆 Skill       · 定义活动规则和审批标准       · 提供权限、安全
  （如活动配置、数据播报）   · 定义资源位优先级规则           日志、监控
· 沉淀模板和交互规范        · 持续优化 Skill 规则           回滚、灰度
                           · 把经验封装为新的 Skill        · 让 Skill 可稳定运行
```

**为什么这样更重要：**
平台运营的需求碎片化、变化快——每次大促规则不同、资源位需求不同、数据口径可能调整。
如果每个场景都靠产品排期开发，一定跟不上业务节奏。
真正可扩张的是"运营自己定义规则、Skill 自动执行"的机制。

### 需要的底座支持

| 能力 | 说明 | 用途 |
|------|------|------|
| 活动配置 API | 满减/券/秒杀等活动类型的配置接口 | 活动配置自动化 |
| 资源位管理 API | 资源位排期查询、配置、上下线接口 | 资源位智能排期 |
| 数据查询 API | GMV、转化率、品类数据、活动效果数据 | 数据播报与复盘 |
| 商品/商家数据 API | 商品池、商家分层、历史提报记录 | 商品筛选与提报 |
| 消息通知接口 | 企微/站内信推送 | 跨团队协作、行动项推送 |
| OpenClaw Skills 框架 | 开发、测试、部署、监控 | 全部场景 |

## Slide 12 — 怎么衡量成效 + 怎么控风险

### 核心指标

| 维度 | 指标 | 目标 |
|------|------|------|
| **活动配置效率** | 单次活动从提报到上线耗时 | 降低 60% |
| **配置质量** | 配置错误率 | 降低 80% |
| **资源位协调效率** | 资源位排期从需求到确认耗时 | 从 2-3 天 → 半天 |
| **资源位冲突率** | 需求冲突导致的返工次数 | 降低 50% |
| **数据分析效率** | 复盘报告产出时间 | 从 1-2 天 → 实时 |
| **行动落地率** | 数据建议→实际执行的转化率 | 提升至 70% |
| **经营结果** | 活动 ROI、资源位转化率 | 持续优化 |

### 风险边界

- **高风险动作留人工**：涉及价格修改、大额满减、资源位首页位等高影响操作，必须人工确认
- **自动执行要有保障**：权限校验、灰度发布、操作日志、一键回滚
- **跨团队边界清晰**：Skill 自动化的是执行层，策略和决策仍由各团队负责人把控
- **数据口径一致**：自动生成的报告和建议，数据来源和计算逻辑必须可追溯

### 我建议的节奏

- **0-3 个月**：日常数据自动播报 + 活动配置助手（L1），先拿效率结果
- **3-6 个月**：活动效果自动复盘 + 资源位排期建议（L2），形成分析能力
- **6-12 个月**：活动全流程自动化 + 资源位自动配置（L3），走向半自动运营

## Slide 13 — 结论

**平台运营最值得做的，不是一个会回答问题的助手，
而是一套能把配置、协调、数据搬运逐步自动化的 OpenClaw 能力体系。**

三大职责的终态：

> **活动配置** — 运营只需定义规则，从提报到配置到上线，数字员工自动完成
> **资源位管理** — 需求汇总、冲突识别、排期推荐、配置上线，全链路自动化
> **数据追踪** — 数据自动播报、效果自动归因、行动建议自动生成并推送执行

最终，运营的核心工作从"搬砖执行"变成"制定规则、审批决策、优化策略"。
