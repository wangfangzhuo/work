## 二、开发做了什么（技能包底座）

开发同学（叶金钢）在 1 周内完成了技能包的底座，对接了 OpenClaw 平台和后台接口。

### 技能包结构

```
pc-home-carousel/
├── skill.json    # 技能元信息（名称、版本、触发入口、MCP 协议声明）
├── package.json  # Node.js 包配置（依赖 MCP SDK、axios）
├── index.js      # 核心逻辑（接口调用）
└── skill.md      # 行为规则文件（初版为空壳，PM 侧补充）
```

### 各文件职责

**skill.json** — 告诉 OpenClaw 平台这个技能叫什么、怎么启动：
```json
{
  "name": "PC首页轮播图管理",
  "code": "pc-home-carousel",
  "type": "mcp",               // 基于 MCP 协议
  "mcp": { "command": "node", "args": ["./index.js"] }
}
```

**index.js** — 技能的可执行部分，做了三件事：
1. **定义工具入参**：声明 7 个参数（楼层名称、分站编码、图片地址、跳转链接、开始时间、结束时间、企业类型），均为必填
2. **企业类型映射**：内置中文名称 → 接口编码的对照表（如「单体药店」→ `3*单体药店`）
3. **调用后台接口**：POST 到 `https://home.yaoex.com/index_pc/addIndexPC`，携带操作者身份（从 OpenClaw session 中读取），返回创建结果

**核心约束**：index.js 要求 7 个参数**全部由调用方提供**，没有任何默认值或推断逻辑。这是技能包能跑通的最小实现。

### 技术选型

| 组件 | 说明 |
|------|------|
| MCP 协议 | OpenClaw 平台的技能接入标准，stdio 模式本地运行 |
| Node.js + axios | 轻量级，快速对接现有后台 HTTP 接口 |
| skill.md | OpenClaw 读取此文件注入给大模型，控制 AI 的行为逻辑 |

### 最新更新（v1.1，叶金钢）

针对权限校验的前置需求，开发完成了两项更新：

**1. 操作者身份识别**：从 OpenClaw 的 session context 中多路径读取 `userId`（兼容 sender_id / user_id / user.id 等不同字段格式），确保任何触发方式下都能正确获取到操作人身份。

**2. 操作日志落地**：所有调用记录写入 `/tmp/yejg-skill.log`，包含 userId、完整请求参数、时间戳，为后续审计和问题排查提供依据。

**当前状态**：身份识别已就绪，但权限校验环节**尚未实现**（见下方安全风险说明）。

## ⚠️ 当前安全风险与待办

### 风险描述

当前技能包**没有权限校验**。任何获得该技能包安装权限的人，都可以直接触发 PC 首页轮播图的新增操作，无需经过任何审批或权限验证。

**这意味着**：一旦技能包被安装到不受控的账号上，或者 OpenClaw 平台的访问控制出现漏洞，内部系统（PC 首页配置）将面临被未授权操作的风险。

### 完整的调用链应该是这样的

```
用户发起对话
    ↓
技能包从 session 中获取 currentUserId（已完成）
    ↓
调用【权限校验 MCP 服务】（待 IT 支持）
    ├── ✅ 有权限 → 展示配置清单 → 用户确认 → 调用创建接口
    └── ❌ 无权限 → 返回「您没有该操作权限，请联系管理员申请」，终止流程
```

### 待 IT 团队提供的能力

需要 IT 团队提供一个**权限校验 MCP 服务**，接受以下输入，返回是否有权限：

| 参数 | 说明 |
|------|------|
| `userId` | 操作人 ID（技能包已可提供） |
| `action` | 操作类型（如 `pc_carousel_create`） |

返回值：`{ "allowed": true/false, "reason": "..." }`

该服务落地后，开发侧在 index.js 中加入权限校验调用即可闭环。**在此之前，技能包不应分发给非受控账号使用。**

## 三、我做了什么（PM 侧）

开发交付后，skill.md **只有基础描述，没有业务规则**，运营必须把所有技术参数一字不差地说出来才能触发创建。

我在此基础上补充了 skill.md 的业务规则层，具体做了三件事：

### 1. 参数智能补全（让运营说人话，不说接口参数）

| 场景 | 原来 | 现在 |
|------|------|------|
| 分站编码 | 必须输入 `000000` | 未提及默认填「默认分站」 |
| 楼层名称 | 必须自己起名 | 未提及时 AI 根据活动内容自动生成（10字以内） |
| 企业类型 | 必须输入数字编码（如 `3,4,8`） | 说「除纯批发」「连锁」「全部」即可，AI 自动映射到对应编码 |
| 生效时间 | 必须写 `2026-05-01 00:00:00` 格式 | 说「五一法定假期那五天」，AI 自动解析为精确时间段 |

### 2. 人机确认环节（防错）

调用接口前，AI 将所有配置项用**用户友好的格式**列出，让运营确认无误后再提交。内部映射的数字编码不对用户暴露，只展示中文业务语言。

```
📋 轮播图配置清单
| 配置项     | 内容                              |
| 楼层名称   | 汤臣倍健五一特惠                    |
| 生效周期   | 2026-05-01 ~ 2026-05-05 五天      |
| 可见企业   | 非批发全部类型（7种）               |
| 跳转链接   | https://mall.yaoex.com/          |
```

### 3. 完整业务规则沉淀

将企业类型映射表、时间解析规则、默认值逻辑全部写入 skill.md，**业务规则成为可版本管理的资产**，后续调整不依赖开发。

## 六、核心判断

这个项目验证了一件事：**AI 落地的关键不是接口，是业务规则层**。

开发同学打通了系统连接（1 周），但没有业务规则的 AI 只是一个严格的表单。PM 用 skill.md 把「运营的经验和判断」编码进去后，才真正让自然语言操作成为可能。

这也意味着后续的迭代主要由 PM 侧驱动：补充规则、优化话术、扩展资源位类型，**不需要每次都拉开发**。
