# OpenClaw 分享会 — 演讲词

---

## 第一部分：理解 OpenClaw

大家好，我今天分享的是平台运营这条线。

在讲场景之前，我想先说一下我对 OpenClaw 的理解。

OpenClaw 不是一个聊天机器人。很多人一听 AI，第一反应就是"能聊天"——但聊天解决不了我们运营的问题。

我理解 OpenClaw 的核心是三层结构：最上面是 Skill，就是一个个封装好的能力单元，比如"活动商品自动审批"、"资源位排期推荐"；中间是 Agent，负责理解你想干什么，然后调用合适的 Skill；最下面是我们的内部系统和数据。

它真正厉害的地方在哪？五个字：**能直接干活**。

它能直连我们的后台系统，不只是告诉你答案，而是直接帮你操作——配置满减、审批提报、推送消息，这些都能做。而且它跟我们的权限体系是打通的，谁能做什么，和现在一样可控。

最关键的是，我们可以把运营的业务规则、审批标准、SOP 封装进 Skill 里，变成可执行的规则。这意味着运营经验不再只存在某个人的脑子里，而是沉淀成了公司的资产。

## 第二部分：三大职责与 Skill 化全景

说回平台运营，我们日常就是三件事：

**第一，活动发布与配置。** 每次大促，配合 JBP、MP、采购各团队做商品提报，然后逐条审批、逐个配置满减。这个过程大家都很熟悉——反复沟通、手动操作、每次大促重复一遍。

**第二，资源位管理。** 资源位类型多、需求方多，排期基本靠 Excel 加群里协商，冲突了就反复调整。

**第三，数据追踪。** 手动拉数据、做表格、写分析，耗时长，而且质量很依赖个人能力。

这三件事有一个共同的问题：**我们 70% 以上的时间都在搬砖——配置、协调、搬数据。** 真正该做的策略思考和规则优化，反而没时间做。

这三个方向都可以 Skill 化，具体有哪些场景大家可以看这张全景表，我就不一个个念了。接下来我挑一个最痛的场景详细说。

---

## 第三部分：重点场景——活动配置自动化

我选的是**活动配置自动化**，因为这是我们每天最耗时、也最适合讲清楚 OpenClaw 价值的场景。

现在的流程大家都经历过：运营策划完活动，先在群里通知各团队提报商品，然后等——等提报、等确认、反复沟通。提报完了逐条审批，不合规的手写理由驳回。审批过了再到后台，逐个商品配置满减、打标。配完了还要人工检查一遍有没有配错。上线之后继续人工盯数据。

**每次大促都重复这套流程，每次都靠人力堆。**

如果用 OpenClaw 呢？运营只需要做一件事：**定义规则**。比如这次活动的门槛是什么、满减机制是什么、目标 GMV 是多少。

剩下的事情 Skill 来做：按规则自动筛选商品，推送给各团队确认；各团队提报自动汇总，合规的秒批，驳回的自动附上整改建议；审批过的商品直接调后台 API 批量完成配置；上线前自动校验，有问题的标出来让运营确认；上线后自动巡检，异常主动预警。

大家看这个对比表就很清楚——每个环节从人工操作变成了 Skill 自动执行。运营的角色从"搬砖执行者"变成了"规则制定者"。

## 第四部分：产品支持模式的根本转变

最后这部分我觉得是最重要的。OpenClaw 不只是一个提效工具，它改变的是**产品支持业务的方式**。

现在是什么情况？业务有新需求，比如新增一种审批规则，得提需求给产品，产品写 PRD，开发排期，开发测试上线，至少 2 到 4 周。下次规则变了？再来一遍。产品和开发疲于奔命，业务也等不及。

OpenClaw 的模式不一样。产品和技术做一件事：**搭建 1 到 2 个基础 Skill，跑通完整流程。** 之后分两条路走：

业务能自己做的——改审批规则、调筛选条件、更新话术模板——运营直接在 Skill 配置里改，当天生效，不需要产品和开发介入。

业务做不了的——比如需要对接一个新的活动类型 API——产品提给技术，技术封装成底座能力。封装完了，运营又可以基于新能力自己组装 Skill。

所以三个角色的分工很清晰：**产品搭标杆、技术封底座、运营自己迭代。** 产品做的是一次性的事，技术是按需封装，运营才是高频使用和持续优化的人。

这样做的好处是什么？新规则上线从 2 到 4 周变成当天生效。产品不用疲于实现碎片化需求，聚焦底座就好。开发封装一次多次复用。最关键的是，运营经验沉淀在 Skill 规则里，人走了能力还在。

**一句话总结：产品从"帮业务实现每一个需求"变成"帮业务搭建自己解决问题的能力"。**
