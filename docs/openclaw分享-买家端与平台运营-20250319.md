## Slide 2 — 分享结构

**这次分享，不是讲"能不能做 AI"，而是讲"先做什么、为什么"**

| 模块 | 内容 |
|------|------|

**一句话结论：**
买家端最好的切入口不是"帮用户搜药"，而是"帮药店老板把采购成本降下来、把比价时间省出来"。
平台运营最好的切入口不是"回答商家问题"，而是"把重复性审核、配置、沟通从运营手里接过来"。

## Slide 3 — 为什么买家端和平台运营值得优先做

**天然具备 Agent / Skill 化条件**

### 买家端

| 特点 | 说明 | 适配度 |
|------|------|--------|
| **痛点明确** | 采购员每天在 4-5 个平台反复搜索比价，耗时 2-3 小时，且容易产生利益冲突 | 极高 |
| **动作重复** | 搜药→比价→选规格→看库存→加车，每次采购都是同一套流程 | 高频 |
| **数据可用** | 各平台的药品价格、库存、效期、优惠都是公开可获取的结构化数据 | 强 |
| **可逐步闭环** | 先做比价展示，再做自动选品，最终走向全自动补货下单 | 强 |

### 平台运营

| 特点 | 说明 | 适配度 |
|------|------|--------|
| **重复性高** | 活动审批、商家邀约、配置满减、问题解答，大量 SOP 化工作 | 极高 |
| **规则明确** | 活动门槛、报名条件、审批标准都有清晰的业务规则 | 强 |
| **系统可连** | 活动配置、商品管理、商家通讯都有现成后台系统 | 中高 |
| **效果可量化** | 处理时长、人均覆盖商家数、审批效率都可直接衡量 | 强 |

**思考重点：**
OpenClaw 不一定从链路最前端开始，而要从"痛点足够密 + 结果可量化 + 数据可获取"的节点切入。
所以第一批场景，优先选择**跨平台比价**和**数据汇总消歧**——这是用户最痛、价值最直接的环节。

## Slide 7 — 重点场景 A：跨平台采购比价（买家端核心）

**这是最容易被理解、也最能直接产出业务价值的场景**

```
用户输入通用名（如"阿莫西林"）
    ↓
[工程化服务] 多平台并行爬取（1药城、药师帮、小药药、药九九）
    ↓
[Skill] 数据汇总 — 按「厂家+规格」聚合归类
    ↓
[Skill] 消歧引导 — 展示候选卡片，用户点击确认
    ↓
[Skill] 比价展示 — 跨平台价格排序，标注最低价、库存、效期、优惠
    ↓
[Skill] 采购决策 — 辅助加车 / 导出采购计划
```

### 架构设计：Skill 智能层 + 工程化服务层

```
┌──────────────────────────────────────────┐
│  Skill（AI 智能层）                       │
│  理解意图 → 汇总归类 → 引导消歧 → 比价展示 │
├──────────────────────────────────────────┤
│  工程化服务（后端）                        │
│  账号管理 / 平台登录 / 数据爬取 / 数据归档  │
└──────────────────────────────────────────┘
```

**关键设计决策：**
- Skill 只负责 AI 智能层，不处理登录和爬取
- 消歧发生在爬取之后（基于真实数据），而非搜索前猜测
- 单 Skill 内含 4 个模块，上下文天然共享，不需要跨 Skill 传参

- ✅ V6 交互 Demo 已完成（Vercel 在线可访问）
- ✅ Skill 架构设计已完成（单 Skill + 4 模块 + 工程化服务）
- ✅ 工程化服务接口已定义（爬取、授权、加车）
- 🔲 后端爬取服务开发中（林超/小白）

## Slide 8 — 重点场景 B：活动运营自动化（平台运营核心）

**运营最痛的就是大促：手动邀约、手动审批、手动配置，全是重复劳动**

```
诊断              方案             配置              巡检             复盘
━━━━━━━━         ━━━━━━━━        ━━━━━━━━         ━━━━━━━━        ━━━━━━━━
历史活动表现       自动给出机制      预填报名参数       活动中监控异常    产出复盘报告
识别适合参与的     建议（满减/券     预算、门槛         及时提醒调价     沉淀下次模板
商家/商品         /秒杀/会场）     商品池             /补货
```

### 具体能力

| 能力 | 当前（人工） | OpenClaw 之后 |
|------|------------|--------------|
| 商家邀约 | 运营手动筛选、逐个发企微 | Skill 按规则自动筛选+批量发送 |
| 报名审批 | 人工逐条审核，驳回手写理由 | 合规的秒批，驳回自动附整改建议 |
| 满减配置 | 手动在后台逐个商品配置 | Skill 调用 API 自动完成配置 |
| 活动巡检 | 大促期间运营盯数据看板 | Skill 自动巡检，异常主动推送 |
| 效果复盘 | 手动导数据、做表格 | Skill 自动生成复盘报告 |

### 想要做深，关键不是大模型，而是 4 类数据
- 活动规则与历史配置
- 商品池 / 价格 / 毛利 / 库存
- 活动效果数据（GMV、转化率、ROI）
- 商家分层与经营目标

## Slide 9 — 重点场景 C/D：运营客服分流 + 智能补货提醒

**一个是规模化效率底座，一个是买家端的演进方向**

### C. 运营客服分流

- 商家在活动提报、配置等环节产生大量咨询
- Skill 自动识别问题类型：活动规则、商品审核、结算、处罚、物流
- 标准问题直接回复 + 连接业务系统操作
- 棘手问题自动上升给运营管理者
- 每次处理后 Skill 自动沉淀到知识库，下次直接命中

**价值：** 减少运营 60%+ 的重复回复，商家响应从小时级降到秒级

### D. 智能补货提醒（买家端演进）

- 接入药店库存 SaaS，监控库存水位
- 触发补货阈值时，自动发起全网比价
- 推送结果到药店老板微信："感冒灵库存剩 20 盒，预计 3 天售罄，已比价 4 平台，最优方案节省 ¥215"
- 药店老板确认即下单

**价值：** 从"人找药"变成"药找人"，买家端从工具升级为服务

## Slide 10 — 怎么排优先级

**不是谁最酷，而是谁最先跑出结果**

用"业务价值 × 数据可用 × 接入难度 × 闭环潜力"来排：

```
                        业务价值 / 闭环潜力 →
                   ┌─────────────────────────────────┐
                   │                                 │
  数据可用         │   运营客服分流    跨平台比价 ★    │  高价值/易落地
  接入容易         │                 活动运营自动化    │  → 优先开做
       ↓          │                                 │
                   ├─────────────────────────────────┤
                   │                                 │
                   │   运营规则沉淀    智能补货提醒    │  高价值/难落地
                   │                 采购计划生成     │  → 作为二期
                   │                                 │
                   └─────────────────────────────────┘
```

**第一批建议：**

| 定位 | 场景 | 理由 |
|------|------|------|
| **标杆场景** | 跨平台采购比价 | 有 Demo、有 Skill 设计、痛点最直接、已在推进 |
| **效率底座** | 运营客服分流 | 高频、规则明确、能快速看到效率提升 |
| **标杆场景** | 活动运营自动化 | 运营最痛的场景、闭环潜力强 |

## Slide 11 — 落地方法论：从"工具"到"自动化"

**分 4 个层级推进，每升一层都更有价值**

```
L1 工具        →    L2 诊断         →    L3 代办          →    L4 闭环
━━━━━━━━            ━━━━━━━━             ━━━━━━━━              ━━━━━━━━
能搜索比价           能解释为什么           能自动配置             能在授权下
能回答价格           这个药贵/便宜          预填表单               自动执行采购
                    定位异常平台           推送补货提醒            自动审批配置
```

### 买家端演进路径

| 阶段 | 能力 | 对应层级 |
|------|------|---------|
| **当前 V1** | 搜索触发 → 比价展示 → 辅助加车 | L1 工具 |
| **V2** | 历史价格趋势、最佳采购时机建议 | L2 诊断 |
| **V3** | 自动组合满减、预填采购单、推送审批 | L3 代办 |
| **终态** | 监控库存 → 自动比价 → 自动下单 → 向老板汇报 | L4 闭环 |

### 平台运营演进路径

| 阶段 | 能力 | 对应层级 |
|------|------|---------|
| **当前** | 回答活动规则、操作指引 | L1 解答 |
| **V2** | 识别适合参加活动的商家、给出邀约建议 | L2 诊断 |
| **V3** | 自动发邀约、预填配置、自动审批简单提报 | L3 代办 |
| **终态** | 全自动活动运营：邀约→审批→配置→巡检→复盘 | L4 闭环 |

## Slide 12 — 怎么衡量成效 + 怎么控风险

### 核心指标

**买家端：**
- 效率：单次比价耗时（目标：从 30 分钟 → 1 分钟）
- 成本：采购成本节省率（目标：平均节省 5-10%）
- 覆盖：接入平台数、比价药品覆盖率
- 粘性：用户留存率、比价后加车转化率

**平台运营：**
- 效率：单次活动配置耗时、商家咨询处理时长
- 覆盖：人均覆盖商家数提升
- 质量：审批准确率、配置错误率下降
- 经营：活动报名率、活动 ROI

### 风险边界

- **采购金额确认**：涉及金钱的加车/下单动作，必须留用户确认环节，不能全自动
- **平台反爬风险**：爬取服务需要控制频率、做好异常处理，避免用户账号被封
- **价格数据准确性**：比价结果必须标注数据查询时间，过期数据要提示刷新
- **授权安全**：用户平台账号密码由工程化服务加密存储，Skill 层不接触明文

### 我建议的节奏

- **0-3 个月**：跨平台比价 V1 上线（已有 Demo），运营客服分流试点
- **3-6 个月**：接入补货提醒，活动运营半自动化
- **6-12 个月**：沉淀 Skill 模板，买家端走向 L3/L4

## Slide 13 — 结论

**买家端最值得做的，不是一个会搜药的工具，
而是一套能把采购动作逐步自动化的 OpenClaw 能力体系。**

**平台运营最值得做的，不是一个会回答问题的客服，
而是一套能把重复性运营动作交给 Skill 执行的自动化机制。**

两条线的共同终态：
> 买家端 — 药店老板不需要自己比价，数字采购员帮他把钱省到位
> 平台运营 — 运营同学不需要搬砖配置，专注于制定规则和策略
